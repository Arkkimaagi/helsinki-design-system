# Buttons

Buttons are meant to make actions easily visible and understandable to the user. HDS offers different kinds of button variations which each suit for different needs.

## Accessibility

- It is advisable to use recommended button color combinations. These combinations are ensured to comply with WCAG AA requirements.
- If using other than recommended button color combinations, remember to take accessibility (especially contrast), general usability and brand guidelines, into account. Related WCAG 2.1 guidelines to check compliance with are 1.4.1 Use of Color and 1.4.3 Contrast (Minimum).

## How to use
- Buttons are used to trigger an action. Be cautious when using buttons for navigating. In most cases you should prefer links for this purpose.Â 
- Button label should always tell the action that the buttons is going to trigger. A good practice is to start the label with a verb and use two word labels at maximum.
- Use provided button types to control the visual priority of the view. Priority of the button types is the following: Primary -> Secondary -> Supplementary.
- Try to keep the amount of buttons in one view low. If there is a need for several actions in the view, consider using the smaller button variant.

### Use of color
Provided recommended button color variations have been designed to be the easy and safe choice for most projects. However, if you feel there is a need to use colors outside of recommended ones, you are allowed to do so. When designing your own button color variations, remember to take accessibility (especially contrast), general usability and brand guidelines into account. Related WCAG 2.1 guidelines to check compliance with are 1.4.1 Use of Color and 1.4.3 Contrast (Minimum).

### Size
In phone width, use full-width buttons. In other sizes use buttons that scale according to their content.

## Primary button
Primary button is reserved for the most important action on the screen. Primary action is usually either mandatory or essential for the user. Primary buttons are designed to clearly highlight the most important action and therefore you should avoid having multiple primary buttons in one screen. For less important actions, consider using secondary or supplementary buttons instead.

</Playground>

##### Core:
```html
<button type="button" class="hds-button hds-button--primary">
  <span class="hds-button__label">Primary</span>
</button>
```

##### React:
```tsx
<Button>Primary</Button>
```

## Secondary button
Secondary buttons are used for actions which are not mandatory or essential for the user. Often screens will include multiple secondary buttons alongside one primary button.

<Playground>
  <Button variant="secondary">Secondary</Button>
</Playground>

##### Core:
```html
<button type="button" class="hds-button hds-button--secondary">
  <span class="hds-button__label">Secondary</span>
</button>
```

##### React:
```tsx
<Button variant="secondary">Secondary</Button>
```

## Supplementary button
Supplementary buttons can be used in similar cases as secondary buttons. However, supplementary buttons are meant for actions which are intentionally wanted to be less visible to the user. These kind of actions include i.e. cancel and dismiss functionalities.

<Playground>
  <Button variant="supplementary">Supplementary</Button>
</Playground>

##### Core:
```html
<button type="button" class="hds-button hds-button--supplementary">
  <span class="hds-button__label">Supplementary</span>
</button>
```

##### React:
```tsx
<Button variant="supplementary">Supplementary</Button>
```

## Utility buttons
If required to achieve clearer user interface, you may also use additional utility colors. Different visual styles of these buttons can be used to better inform users of destructive or dangerous actions. To comply with WCAG requirement 1.4.1 Use of Color, these colors are often better when paired with an icon.

    <br />
